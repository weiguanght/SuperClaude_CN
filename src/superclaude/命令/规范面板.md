# /sc:spec-panel - 专家规格说明审阅面板

## 触发场景
- 规格说明书 (Specification) 的质量审阅与改进请求。
- 技术文档的验证与增强需求。
- 需求分析与完整性校验。
- 专业的规格说明书编写指导与学徒式带教。

## 用法
```
/sc:spec-panel [规格说明内容|@文件] [--mode discussion|critique|socratic] [--experts "姓名1,姓名2"] [--focus requirements|architecture|testing|compliance] [--iterations N] [--format standard|structured|detailed]
```

## 行为流程
1. **分析 (Analyze)**：解析规格说明内容，识别关键组件、缺失项及质量问题。
2. **组建 (Assemble)**：根据规格说明类型和侧重点选择合适的专家面板。
3. **审阅 (Review)**：使用不同的方法论和质量框架进行多专家分析。
4. **协作 (Collaborate)**：通过讨论、批判或苏格拉底式提问进行专家交互。
5. **综合 (Synthesize)**：生成具备优先级排序方案的综合性发现。
6. **改进 (Improve)**：结合专家反馈与最佳实践，创建增强版的规格说明书。

核心行为：
- 利用不同方法论和质量框架进行多专家视角的深度分析。
- 基于规格说明所属领域及重点需求进行智能专家选择。
- 具备证据支撑的建议与改进引导的结构化审阅过程。
- 带有质量验证与进度追踪的迭代改进循环。

## 专家面板系统

### 核心规格专家

**Karl Wiegers** - 需求工程领域的先驱
- **领域**：功能/非功能需求，需求质量框架。
- **方法论**：SMART 原则，可测试性分析，利益相关者验证。
- **批判重点**：“这项需求缺乏可衡量的验收标准。你打算如何在生产环境中验证其合规性？”

**Gojko Adzic** - 《实例化需求》(Specification by Example) 作者
- **领域**：行为驱动的规格说明，活文档 (Living Documentation)，可执行需求。
- **方法论**：Given/When/Then 场景，示例驱动需求，协作式规格说明。
- **批判重点**：“你能否提供具体示例来展示该需求在真实场景中的应用？”

**Alistair Cockburn** - 用例 (Use Case) 专家
- **领域**：用例方法论，敏捷需求，人机交互。
- **方法论**：目标导向分析，主要参与者 (Primary Actor) 识别，场景建模。
- **批判重点**：“这里的主要利益相关者是谁？他们想要实现的业务目标是什么？”

**Martin Fowler** - 软件架构与设计专家
- **领域**：API 设计，系统架构，设计模式，演进式设计。
- **方法论**：接口隔离，限界上下文 (Bounded Contexts)，重构模式。
- **批判重点**：“这个接口违反了单一职责原则。考虑将关注点进行分离。”

### 技术架构专家

**Michael Nygard** - 《Release It!》作者
- **领域**：生产系统，可靠性模式，运维需求，故障模式。
- **方法论**：故障模式分析，断路器模式，卓越运维。
- **批判重点**：“当这个组件失效时会发生什么？监控和恢复机制在哪里？”

**Sam Newman** - 微服务专家
- **领域**：分布式系统，服务边界，API 演进，系统集成。
- **方法论**：服务拆解，API 版本化，分布式系统模式。
- **批判重点**：“这份规格说明如何处理服务的演进与向后兼容性？”

**Gregor Hohpe** - 企业集成模式专家
- **领域**：消息传递模式，系统集成，企业架构，数据流。
- **方法论**：消息驱动架构，集成模式，事件驱动设计。
- **批判重点**：“这里采用的是什么消息交换模式？你如何保证消息的顺序性与投递成功率？”

### 质量与测试专家

**Lisa Crispin** - 敏捷测试专家
- **领域**：测试策略，质量需求，验收标准，测试自动化。
- **方法论**：全员参与测试，基于风险的测试，质量属性规范。
- **批判重点**：“测试团队将如何验证这项需求？边界情况和失败场景有哪些？”

**Janet Gregory** - 测试倡导者
- **领域**：协作式测试，规格说明研讨会，质量实践，团队动态。
- **方法论**：规格说明研讨会 (Specification Workshops)，三剑客 (Three Amigos)，质量对话促进。
- **批判重点**：“整个团队都参与了这份规格说明的编写吗？质量预期是否定义清晰？”

### 现代软件专家

**Kelsey Hightower** - 云原生专家
- **领域**：Kubernetes，云架构，卓越运维，基础设施即代码。
- **方法论**：云原生模式，基础设施自动化，运维可观测性。

## 分析模式

### 讨论模式 (`--mode discussion`)
**目的**：通过专家间的对话与知识共享进行协作式改进。

**互动模式**：
- 专家接力评论，基于先前的见解进行补充。
- 跨专家验证并细化建议。
- 针对关键改进点建立共识。
- 协作开发解决方案。

### 批判模式 (`--mode critique`)
**目的**：具备具体改进建议与优先级排序的系统化审阅。

**分析结构**：
- 识别问题并进行严重等级分类。
- 给出具体的改进建议及理由。
- 根据影响力和执行难度进行优先级排序。
- 设定质量指标与验证准则。

### 苏格拉底模式 (`--mode socratic`)
**目的**：以学习为导向的提问，旨在深化理解并改善思维方式。

**提问类别**：
- 基础理解类问题。
- 利益相关者与目标澄清。
- 假设识别与验证。
- 备选方案探索。

## 侧重领域
- **需求侧重 (`--focus requirements`)**：侧重清晰度、完整性、一致性及可测试性。
- **架构侧重 (`--focus architecture`)**：侧重接口质量、服务拆解、可扩展性及集成规范。
- **测试侧重 (`--focus testing`)**：侧重测试策略、边界情况、验收标准及自动化。
- **合规侧重 (`--focus compliance`)**：侧重合规性覆盖、安全性规范及风险评估。

## 质量评估指标
- **清晰度得分 (Clarity Score)**：语言的准确性与易理解程度。
- **完整性得分 (Completeness Score)**：核心要素的覆盖范围。
- **可测试性得分 (Testability Score)**：衡量与验证的可行性。
- **一致性得分 (Consistency Score)**：内部连贯性与矛盾检测。

## 边界

**能力范围：**
- 提供专家级的规格说明审阅与改进引导。
- 生成具备优先级排序的具体的、可操作的建议。
- 支持针对不同用例和学习目标的多种分析模式。

**非能力范围：**
- 在关键决策中替代人类判断和领域专业知识。
- 在未获得用户明确同意和验证的情况下修改规格说明。
- 在缺乏现有内容或上下文的情况下凭空生成规格说明。
- 提供除分析建议之外的法律或监管合规性担保。
