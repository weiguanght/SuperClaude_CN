# /sc:design - 系统与组件设计

## 触发场景
- 架构规划与系统设计请求。
- API 规格说明与接口设计需求。
- 组件设计与技术规格要求。
- 数据库模式 (Schema) 与数据模型设计请求。

## 用法
```
/sc:design [目标] [--type architecture|api|component|database] [--format diagram|spec|code]
```

## 行为流程
1. **分析 (Analyze)**：检查目标需求及现有系统上下文。
2. **计划 (Plan)**：根据类型和格式定义设计方案与结构。
3. **设计 (Design)**：结合行业最佳实践创建详尽的规格说明。
4. **验证 (Validate)**：确保设计满足需求及可维护性标准。
5. **文档 (Document)**：生成包含图表和规格说明的清晰设计文档。

核心行为：
- 需求驱动的设计方法，并充分考虑可扩展性。
- 集成行业最佳实践以提供可维护的解决方案。
- 根据需求提供多格式输出（图表、规格说明、代码）。
- 针对现有系统架构和约束进行验证。

## 工具协作
- **Read**：需求分析与现有系统检查。
- **Grep/Glob**：模式分析与系统结构调研。
- **Write**：设计文档与规格说明生成。
- **Bash**：必要时进行外部分析工具集成。

## 关键模式
- **架构设计**：需求 → 系统结构 → 可扩展性规划。
- **API 设计**：接口规范 → RESTful/GraphQL 模式 → 文档化。
- **组件设计**：功能需求 → 接口设计 → 实施指南。
- **数据库设计**：数据需求 → 模式设计 → 关系建模。

## 示例

### 系统架构设计
```
/sc:design 用户管理系统 --type architecture --format diagram
```
# 创建包含组件关系的综合系统架构
# 包含可扩展性考量与最佳实践

### API 规格说明设计
```
/sc:design 支付 API --type api --format spec
```
# 生成包含端点和数据模型的详细 API 规格说明
# 遵循 RESTful 设计原则与行业标准

### 组件接口设计
```
/sc:design 通知服务 --type component --format code
```
# 设计具有明确契约与依赖关系的组件接口
# 提供实施指南与集成模式

### 数据库模式设计
```
/sc:design 电商数据库 --type database --format diagram
```
# 创建包含实体关系与约束的数据库模式
# 包含范式化与性能考量

## 边界

**能力范围：**
- 结合行业最佳实践创建全面的设计规格说明。
- 根据需求生成多种格式的输出（图表、规格、代码）。
- 针对可维护性与可扩展性标准验证设计。

**非能力范围：**
- 生成实际的项目实施代码（请使用 `/sc:implement` 进行实施）。
- 在未获得明确设计批准的情况下修改现有系统架构。
- 创建违反既定架构约束的设计。
