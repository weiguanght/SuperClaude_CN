# /sc:estimate - 开发估算

## 触发场景
- 需要时间、工作量或复杂度估算的开发规划。
- 项目范围界定与资源分配决策。
- 需要系统化估算法的功能拆解。
- 风险评估与置信区间分析需求。

## 用法
```
/sc:estimate [目标] [--type time|effort|complexity] [--unit hours|days|weeks] [--breakdown]
```

## 行为流程
1. **分析 (Analyze)**：检查范围、复杂度因素、依赖项以及框架模式。
2. **计算 (Calculate)**：应用包含历史基准和复杂度评分的估算法。
3. **验证 (Validate)**：结合项目模式和领域专业知识交叉印证估算结果。
4. **展示 (Present)**：提供带有置信区间和风险评估的详细拆解。
5. **追踪 (Track)**：记录估算准确度，以便持续改进估算法。

核心行为：
- 根据估算范围协调多个人格 (架构师、性能工程师、项目经理)。
- 使用 Sequential MCP 进行系统化分析与复杂度评估。
- 使用 Context7 MCP 获取框架特定的模式与历史基准数据。
- 具备置信区间和风险因素的智能拆解分析。

## MCP 集成
- **Sequential MCP**：用于复杂的、多步骤的估算分析及系统化的复杂度评估。
- **Context7 MCP**：提供框架特定的估算模式和历史基准数据。
- **人格协作**：架构师 (设计复杂度)、性能工程师 (优化工作量)、项目经理 (时间线)。

## 工具协作
- **Read/Grep/Glob**：代码库分析以评估复杂度和工作范围。
- **TodoWrite**：针对复杂估算工作流的估算拆解与进度追踪。
- **Task**：针对需要系统化协调的跨领域估算进行高级委派。
- **Bash**：项目分析与依赖评估，以实现准确的复杂度评分。

## 关键模式
- **范围分析**：项目需求 → 复杂度因素 → 框架模式 → 风险评估。
- **估算法**：基于时间、基于工作量、基于复杂度或基于成本的方法。
- **多领域评估**：架构复杂度 → 性能要求 → 项目时间线。
- **验证框架**：历史基准 → 交叉验证 → 置信区间 → 准确度追踪。

## 示例

### 功能开发估算
```
/sc:estimate "用户身份验证系统" --type time --unit days --breakdown
```
# 系统化分析：数据库设计 (2天) + 后端 API (3天) + 前端 UI (2天) + 测试 (1天)
# 总计：8 天，置信区间为 85%

### 项目复杂度评估
```
/sc:estimate "将单体架构迁移至微服务" --type complexity --breakdown
```
# 包含风险因素和依赖映射的架构复杂度分析
# 通过多个人格协调进行的全面评估

### 性能优化工作量
```
/sc:estimate "优化应用性能" --type effort --unit hours
```
# 性能工程师人格分析，并附带基准对比
# 按优化类别和预期影响拆解工作量

## 边界

**能力范围：**
- 提供带有置信区间和风险评估的系统化开发估算。
- 应用多个人格协作进行全面的复杂度分析。
- 生成包含历史基准对比的详细拆解分析。

**非能力范围：**
- 在没有经过适当的范围分析与验证的情况下保证估算的准确性。
- 在缺乏相应领域知识和复杂度评估的情况下提供估算。
- 在没有明确理由和分析支持的情况下覆盖历史基准。
