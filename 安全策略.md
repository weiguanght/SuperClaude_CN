# 安全策略

## 🔒 报告安全漏洞

SuperClaude 框架通过安全设计原则、全面的输入验证和负责任的漏洞管理来优先考虑安全性。我们致力于在实现强大的 AI 辅助工作流的同时维护一个安全的开发平台。

**安全承诺:**
- 及时响应安全报告 (48-72 小时)
- 关于安全问题的透明沟通
- 定期安全审计与依赖项更新
- 社区驱动的安全改进

### 负责任的披露

**主要联系方式:** anton.knoery@gmail.com (由维护者监控)

**流程:**
1. **报告**: 将详细的漏洞报告发送至 anton.knoery@gmail.com
2. **确认**: 我们将在 48 小时内确认收到
3. **调查**: 72 小时内进行初步评估
4. **协调**: 共同开发修复方案并进行测试
5. **披露**: 修复部署后进行协调的公开披露

**备选渠道:**
- GitHub 安全公告 (针对 GitHub 托管的问题)
- 针对关键漏洞直接联系维护者
- 应请求可提供加密通信

**请务必:**
- 提供详细的技术描述和重现步骤
- 为调查和修复开发留出合理时间
- 在协调披露前保持机密性

**请勿:**
- 在协调前公开披露漏洞
- 在您不拥有的系统上测试漏洞
- 超出概念验证演示范围访问或修改数据

### 应包含的内容

**必需信息:**
- SuperClaude 版本: `SuperClaude --version`
- 操作系统及版本
- Python 版本: `python3 --version`
- Claude Code 版本: `claude --version`
- 漏洞描述及潜在影响
- 带最小测试用例的详细重现步骤
- 概念验证代码或命令 (如适用)

**有用的附加详情:**
- 涉及的 MCP 服务器配置 (如适用)
- 网络环境与代理配置
- 自定义行为模式或智能体配置
- 日志文件或错误消息 (已清理个人数据)
- 漏洞演示的截图或录屏

**漏洞报告模板:**
```
**SuperClaude 版本:** [版本]
**环境:** [操作系统, Python 版本, Claude Code 版本]

**漏洞摘要:**
[安全问题的简要描述]

**影响评估:**
[潜在的安全影响及受影响组件]

**重现步骤:**
1. [分步说明]

3. [显示预期 vs 实际行为]

**概念验证:**
[演示问题的最小代码或命令]

**建议修复:**

```

### 响应时间线

**响应时间线:**

**初步响应: 48 小时**
- 确认收到漏洞报告
- 分配内部跟踪标识符
- 提供初步影响评估

**调查: 72 小时**
- 确认漏洞并评估严重性
- 识别受影响的版本和组件
- 开始修复开发规划

**状态更新: 每周**
- 调查期间的定期进度更新
- 如复杂性需要延期则调整时间线
- 协调披露时间线

**修复开发: 取决于严重性**
- **关键**: 7-14 天开发补丁
- **高**: 14-30 天进行全面修复
- **中**: 30-60 天彻底解决
- **低**: 下一个常规发布周期

**披露协调:**
- 公开披露前提前通知报告者
- 安全公告准备与审查
- 与修复部署协调发布
- 公开认可负责任的披露

**紧急响应:**
对于正在被利用的漏洞或关键安全问题:
- 12 小时内立即响应
- 紧急补丁开发与测试
- 加速披露流程并通知社区

## 🚨 严重性级别

**关键 (CVSS 9.0-10.0)**
- **示例**: 远程代码执行、任意文件系统访问、凭据窃取
- **响应**: 12 小时确认, 7 天修复目标
- **影响**: 完全系统妥协或数据泄露可能性

**高 (CVSS 7.0-8.9)**
- **示例**: 权限提升、敏感数据暴露、身份验证绕过
- **响应**: 24 小时确认, 14 天修复目标
- **影响**: 重大安全控制绕过或数据访问

**中 (CVSS 4.0-6.9)**
- **示例**: 信息泄露、拒绝服务、配置操纵
- **响应**: 48 小时确认, 30 天修复目标
- **影响**: 有限的安全影响或特定攻击场景

**低 (CVSS 0.1-3.9)**
- **示例**: 轻微信息泄露、速率限制绕过、非关键验证错误
- **响应**: 72 小时确认, 下一个发布周期
- **影响**: 需要特定条件的最小安全影响

**严重性评估因素:**
- **攻击向量**: 网络可访问 vs 需要本地访问
- **攻击复杂度**: 简单 vs 复杂的利用要求
- **所需权限**: 无 vs 需要经过身份验证的访问
- **用户交互**: 自动 vs 需要用户操作
- **范围**: 框架核心 vs 特定组件影响
- **机密性/完整性/可用性影响**: 完全 vs 部分 vs 无

**特殊考虑:**
- MCP 服务器漏洞基于最坏情况配置进行评估
- 智能体协调问题评估权限提升可能性
- 配置文件漏洞考虑凭据暴露风险

**当前支持的版本:**

| 版本 | 安全支持 | 支持结束日期 |
|---------|------------------|----------------|\n| 4.1.x   | ✅ 完全支持  | 待定 (当前) |

**支持策略:**
- **完全支持**: 所有安全问题均通过定期补丁解决
- **仅关键问题**: 仅关键漏洞 (CVSS 9.0+) 获得补丁
- **不支持**: 无安全补丁; 用户应立即升级

**版本支持生命周期:**
- **当前主版本**: 整个生命周期的完全安全支持
- **上一个主版本**: 新主版本发布后 12 个月的关键安全支持
- **遗留版本**: 无支持; 需要升级以获得安全修复

**安全更新分发:**
- 关键补丁: 立即发布并紧急通知
- 高严重性: 与常规更新周期协调发布
- 中/低: 包含在下一个计划发布中

**升级建议:**
- 始终使用最新稳定版本以获得最佳安全态势
- 订阅安全通知以获取及时更新信息
- 在生产部署前在开发环境中测试更新
- 查看安全公告以进行影响评估

**企业支持:**
对于需要延长安全支持的组织:
- 联系维护者进行定制支持安排
- 考虑为开发做贡献以获得优先处理
- 为不受支持的版本实施额外的安全控制

## 🔧 安全最佳实践

### 用户最佳实践

**安装安全:**
- 仅从官方来源下载 SuperClaude (PyPI, npm, GitHub 发布)
- 验证包签名和校验和 (如可用)
- 使用虚拟环境隔离依赖项
- 保持 Python, Node.js 和系统包更新

**配置安全:**
- 对 ~/.claude 目录使用安全文件权限 (仅用户访问)
- 将 API 凭据存储在系统凭据管理器中,而非配置文件
- 定期审查和审计 MCP 服务器配置
- 仅启用所需的 MCP 服务器以最小化攻击面

**项目安全:**
- 除非绝对必要,否则不要以提升的权限运行 SuperClaude
- 在执行前审查生成的代码,尤其是外部 API 调用
- 使用版本控制跟踪所有 SuperClaude 生成的更改
- 定期备份项目配置和重要数据

**网络安全:**
- 对所有外部 MCP 服务器连接使用 HTTPS
- 使用访问外部 API 的 MCP 服务器时要谨慎
- 在限制性环境中考虑网络防火墙
- 监控网络流量以发现意外的外部连接

**数据隐私:**
- 使用基于云的 MCP 服务器时注意项目文件中的敏感数据
- 查看 MCP 服务器隐私政策和数据处理实践
- 对敏感项目尽可能使用仅本地的 MCP 服务器
- 定期清理临时文件和会话数据

**命令使用:**
- 使用 `--dry-run` 标志预览潜在的破坏性操作
- 在执行前了解命令范围和权限
- 对修改多个文件或系统配置的命令要谨慎
- 在继续依赖操作前验证命令输出和结果

### 安全公告流程
**安全公告生命周期:**

**公告创建:**
1. **初步评估**: 漏洞验证和影响分析
2. **公告草案**: 技术描述、受影响版本和影响评估
3. **修复开发**: 协调补丁开发并进行测试
4. **预发布审查**: 公告准确性和完整性验证

**利益相关者协调:**
- **报告者沟通**: 定期更新和修复验证协作
- **维护者审查**: 技术准确性和修复验证
- **社区准备**: 高影响漏洞的预公告
- **下游通知**: 警报依赖项目和分发

**披露时间线:**
- **协调披露**: 修复可用后 90 天标准时间线
- **紧急披露**: 对于正在被利用的漏洞立即披露
- **延长协调**: 经事先同意,复杂修复的额外时间
- **公开发布**: 修复部署后公告发布

**公告内容:**
- **漏洞描述**: 安全问题的清晰技术解释
- **影响评估**: CVSS 分数和真实世界影响分析
- **受影响版本**: 易受攻击的框架版本的完整列表
- **修复信息**: 补丁详情、变通方法和升级说明
- **致谢**: 负责任披露致谢和研究人员认可

**分发渠道:**
- 主要通知的 GitHub 安全公告
- 社区邮件列表和讨论论坛
- 高影响问题的社交媒体公告
- 正式跟踪的漏洞数据库 (CVE, NVD)

**披露后:**
- 社区问答和公告理解支持
- 经验教训分析和流程改进
- 基于发现问题的安全文档更新
- 类似漏洞类别的增强测试和验证
