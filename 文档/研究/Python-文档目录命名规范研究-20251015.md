# Python 文档目录命名规范研究 (Python Documentation Directory Naming Convention Research)

**日期**：2025-10-15
**研究课题**：Python 项目中文档目录的正确命名规范是什么？
**背景**：SuperClaude 框架上游目前混合使用了 PascalCase (大驼峰) 与小写命名，导致了不一致性。在提出标准化方案前，需明确 Python 生态系统的最佳实践。

---

## 🎯 执行摘要 (Executive Summary)

**调研发现**：Python 生态系统压倒性地使用**全小写**字母命名文档目录，对于多单词目录，通常使用**连字符 (hyphen)** 进行连接。

**核心证据**：调研的 5 个主流 Python 项目（Django, CPython, Flask, FastAPI, Requests）全部采用小写命名方案。

**最终建议**：标准化为“小写+连字符”模式（例如：`user-guide`, `developer-guide`），从而与 Python 的核心哲学与生态系统惯例保持一致。

---

## 📚 官方标准分析

### PEP 8 - Python 代码风格指南
- **包与模块**：应使用简短、**全小写**的名称。
- **下划线**：仅在提高可读性时鼓励使用。
- **启示**：虽然 PEP 8 主要针对源码，但“全小写”是 Python 命名哲学的基石。

### PEP 423 - 发行版命名规范
- **PyPI 发行名称**：推荐使用连字符（如 `my-package`）。
- **导入包名**：使用下划线（如 `my_package`）。
- **启示**：面向用户的目录（如文档）应遵循类似于发行名称的连字符惯例。

## 主流项目分析

| 项目 | 根目录名称 | 子目录示例 | 命名模式 |
| :--- | :--- | :--- | :--- |
| **Django** | `docs/` | `intro/`, `ref/`, `releases/` | 全小写 |
| **CPython** | `Doc/` | `c-api/`, `tutorial/`, `whatsnew/` | 小写 + 连字符 |
| **Flask** | `docs/` | `deploying/`, `errorhandling/` | 全小写 (合并单词) |
| **FastAPI** | `docs/` | `tutorial/`, `en/`, `zh/` | 全小写 |

## SuperClaude 现状与问题

目前上游存在重大的命名不一致性：
- `Developer-Guide/` (大驼峰 + 连字符)
- `development/` (全小写)
- `Getting-Started/` (大驼峰 + 连字符)
- `troubleshooting/` (全小写)

**存在的问题**：
1. **不符合 PEP 8**：违反了全小写原则。
2. **生态疏离感**：大驼峰命名在 Python 社区极少见，更见于 Java 或 C# 项目。
3. **跨平台风险**：大小写不敏感的文件系统（如 macOS/Windows）与敏感系统（Linux）同步时易产生 Git 冲突。

---

## 💡 最终建议

### 首选方案：全小写 + 连字符
**范式**：`lowercase-with-hyphens`

**理由**：
1. **可读性强**：连字符在视觉上优于直接拼写合并。
2. **生态一致**：与 Python 官方 (CPython) 文档结构完美吻合。
3. **工具友好**：与 Sphinx, MkDocs, ReadTheDocs 等文档工具无缝协同。

---

## 🚀 迁移策略建议

**第一步：批量重命名 (git mv)**
将 `User-Guide` 改为 `user-guide` 等。

**第二步：更新引用**
全局更新所有内部 Markdown 链接及 CI/CD 脚本。

**第三步：发布说明**
由于此举会破坏外部旧链接，建议在重大版本（如 v5.0）中实施，并发布公告。

**结论**：PascalCase 在 Python 文档命名中是“异类”。为了专业性与规范性，应尽快统一为小写连字符模式。
